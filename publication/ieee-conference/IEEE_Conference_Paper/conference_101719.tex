\begin{abstract}
Arabic optical character recognition (OCR) systems produce systematic
character-level errors---dot confusions, hamza substitutions, taa marbuta
swaps---that degrade downstream text processing. Post-OCR correction using
large language models (LLMs) is a natural remedy, but it is unclear how much
additional context actually helps, and whether findings generalize across
font families and script types. This paper reports a controlled experiment
in which seven prompt-engineering strategies are evaluated across nine
datasets---eight typewritten fonts from PATS-A01 and one handwritten set from
KHATT---using Qwen3-4B-Instruct-2507 as the correction model. The strategies
tested are: zero-shot correction, confusion-matrix injection, rule-augmented
prompting, few-shot examples from the QALB corpus, morphological
post-processing via CAMeL Tools, self-reflective prompting with
training-split error analysis, optimal strategy combinations with ablation,
and DSPy-based automated prompt optimization. Results reveal that
correction effectiveness is strongly font-dependent: zero-shot LLM correction
reduces CER by up to \tbd\% on high-error typewritten fonts but can increase
errors on fonts where Qaari already performs well. On handwritten text (KHATT),
where baseline OCR errors exceed 40\% CER, the LLM provides marginal
improvement. These findings suggest that the optimal correction strategy
depends on the baseline error profile, and that a one-size-fits-all approach
to prompt augmentation is unlikely to succeed.
\end{abstract}

\section{Introduction}
% -------------------------------------------------------------------------

Arabic optical character recognition is a harder problem than it looks. The
script is cursive, dots are phonemically significant, and a single misplaced
dot can change the meaning of a word entirely. Open-source OCR engines such
as Qaari~\cite{qaari} have made Arabic recognition more accessible, but they
still produce character-level errors that compound in downstream tasks like
information retrieval or machine translation.

Post-OCR correction---treating the OCR output as noisy text and using a language
model to clean it---has seen renewed interest since powerful instruction-tuned
LLMs became widely available. The appeal is obvious: no training data is needed,
the correction can be done in a single forward pass, and the model's language
priors should naturally prefer valid Arabic words. What is less obvious is
whether feeding the model more context actually improves the result, and
whether findings on one font or script type generalize to others.

We set up a controlled experiment to test exactly this. Starting from zero-shot
correction as a baseline, we progressively enriched the prompt with different
knowledge sources: the OCR engine's known confusion patterns, explicit Arabic
spelling rules, error--correction examples from a corpus, the model's own
diagnosed failure modes from training-split analysis, and combinations thereof.
Seven prompt strategies in total, evaluated across nine datasets spanning eight
typewritten fonts and one handwritten corpus, so that font-specific effects can
be separated from strategy effects.

The main finding is nuanced: correction effectiveness depends strongly on the
baseline error rate. On typewritten fonts with moderate-to-high OCR error rates,
zero-shot LLM correction provides substantial CER reductions. On fonts where
Qaari already achieves low error rates, the LLM tends to introduce more errors
than it fixes. On handwritten text, where errors are abundant but diverse, the
LLM provides only marginal improvement. Knowledge augmentation shows
mixed results across fonts, with no single strategy dominating.

The contributions of this work are:
\begin{itemize}
\item A systematic comparison of seven prompt strategies for Arabic post-OCR
      correction across nine datasets, revealing font-dependent behavior that
      single-dataset studies would miss.
\item Evidence that correction effectiveness is determined primarily by the
      baseline OCR error profile, not by the amount of context in the prompt.
\item A novel self-reflective prompting approach that feeds the model its own
      error statistics from training-split analysis.
\item An ablation study of strategy combinations and an evaluation of
      automated prompt optimization via DSPy.
\end{itemize}

\section{Related Work}
% -------------------------------------------------------------------------

\subsection{Arabic OCR and Its Error Patterns}
Arabic OCR has been studied since the 1990s~\cite{arabic_ocr_survey}, and the
core difficulties are well documented: the cursive baseline makes segmentation
hard, diacritics are often omitted or misread, and several letter groups
(ba/ta/tha, dal/dhal, ra/zay) share the same skeleton differing only in dots.
Qaari, the open-source engine used in this study, is based on Tesseract with
Arabic-specific training and inherits both its strengths and its failure
modes~\cite{qaari}.

\subsection{Post-OCR Correction}
Traditional post-OCR correction relied on language models, lexicon lookup, or
a combination of both~\cite{afli2016,dong2018}. The move toward neural
approaches brought sequence-to-sequence models that treat correction as a
translation task~\cite{nguyen2021}. More recently, prompted LLMs have been
tested for English document correction with encouraging
results~\cite{llm_ocr_correction}, but Arabic-specific studies remain sparse.
The most directly relevant prior work uses Arabic BERT variants for post-hoc
correction~\cite{arabic_bert_ocr}, but the prompt-engineering perspective has
not been explored systematically across multiple fonts and script types.

\subsection{Knowledge-Augmented Prompting}
Injection of structured knowledge into LLM prompts has been shown to help in
several settings: few-shot examples improve performance across
tasks~\cite{brown2020}, chain-of-thought prompts help with
reasoning~\cite{wei2022}, and retrieval-augmented generation grounds outputs
in factual content~\cite{lewis2020}. There is also work on using a model's own
error statistics to guide its future behavior, sometimes called self-reflection
or self-critique~\cite{shinn2023}. DSPy~\cite{dspy2024} provides a framework
for automated prompt optimization, compiling declarative language model
programs into optimized prompts. Our study applies these ideas to OCR
correction and measures whether the gains transfer to this domain.

\subsection{Arabic NLP Resources}
The QALB corpus provides Arabic text with corresponding corrections for human
typing errors~\cite{mohit2014}, making it the closest available resource to
OCR correction examples. CAMeL Tools provides morphological analysis and
disambiguation for modern standard Arabic~\cite{obeid2020}, which we use for
post-correction validation.

\section{Datasets and Experimental Setup}
% -------------------------------------------------------------------------

\subsection{Datasets}
All experiments use two complementary Arabic OCR benchmarks.

\textbf{PATS-A01}~\cite{pats} is a typewritten Arabic dataset covering eight
font families: Akhbar, Andalus, Arial, Naskh, Simplified, Tahoma, Thuluth,
and Traditional. Each font subset contains synthetic document images with
known ground-truth text. We use an 80/20 train/validation split per font
(seed=42), yielding 553 validation samples per font. Training splits are used
for self-reflective analysis (Phase~4D); validation splits are used for all
reported metrics.

\textbf{KHATT}~\cite{khatt} is a handwritten Arabic dataset with real-world
variation in writing style, pen type, and image quality. It provides a
significantly harder challenge than typewritten text: baseline OCR error rates
exceed 40\% CER, compared to single-digit CER for most PATS-A01 fonts.

\subsection{OCR Engine}
We use Qaari as the OCR engine. Qaari's output serves as both the baseline
for Phase~1 analysis and the input to all LLM correction phases.

\subsection{Correction Model}
Qwen3-4B-Instruct-2507~\cite{qwen3} is used as the LLM for all correction
phases. It is a 4-billion-parameter instruction-tuned model with strong
multilingual capabilities and native Arabic support. Inference was run on
a GPU-accelerated environment (Kaggle T4). The \texttt{enable\_thinking} flag
was set to \texttt{False} to suppress the chain-of-thought scratchpad, which
increases token cost without measurable benefit for this task.

\subsection{Evaluation Metrics}
Character Error Rate (CER) is the primary metric:
\begin{equation}
\text{CER} = \frac{S + D + I}{N},
\label{eq:cer}
\end{equation}
where $S$, $D$, $I$ are the number of substitutions, deletions, and insertions
at the character level, and $N$ is the total number of characters in the
reference. Word Error Rate (WER) is reported alongside CER for completeness.
Lower values indicate better performance.

\subsection{Pipeline Architecture}
Each experimental phase follows a three-stage pipeline: (1)~\textit{export},
which prepares the inference input JSONL from the local dataset;
(2)~\textit{infer}, which runs the LLM on a GPU-capable environment with
periodic checkpointing to HuggingFace Hub for fault tolerance; and
(3)~\textit{analyze}, which computes metrics and comparison statistics locally.
This separation allows the correction model to run on cloud GPUs while
evaluation happens on the local machine with access to ground-truth data.

\section{Experimental Methodology}
% -------------------------------------------------------------------------

We organize the experiment into eight phases. Phase~1 characterizes the OCR
errors. Phase~2 is the zero-shot LLM correction baseline. Phases~3, 4A--4D
each introduce exactly one form of additional knowledge, isolating the effect
of each augmentation relative to Phase~2. Phase~5 tests combinations of the
most promising strategies with ablation analysis. Phase~7 uses DSPy to
automatically optimize prompts.

\subsection{Phase 1: OCR Baseline and Error Taxonomy}

Before correcting anything, we characterize what there is to correct.
Phase~1 runs Qaari over all datasets and computes CER and WER against
ground truth. It also builds a character-level confusion matrix per dataset
and classifies errors into categories: dot confusion (ba/ta/tha/ya
misreads), hamza errors, taa marbuta/ha swaps, alef-maksura/ya swaps,
similar-shape pairs (ra/zay, dal/dhal), and other substitutions.

Morphological validity of OCR tokens is checked with CAMeL
Tools~\cite{obeid2020}. Words that fail morphological analysis are classed as
non-word errors; those that parse successfully but differ from the ground
truth are valid-but-wrong errors. This distinction is important because
non-word errors should be straightforward for an LLM to catch, whereas
valid-but-wrong errors require contextual understanding.

\subsection{Phase 2: Zero-Shot LLM Correction (Baseline)}

The zero-shot prompt instructs the model to correct OCR errors in the supplied
text and return only the corrected output without explanation. The system prompt
uses an XML-structured template specifying the task, nine correction rules
(preserve meaning, fix visual OCR errors only, do not add diacritics, etc.),
and few-shot examples demonstrating the expected input--output format. This
conservative formulation was chosen to reduce unnecessary rewrites, which are
a known failure mode for instruction-following models on text that is mostly
correct.

Phase~2 output serves as the comparison point for all subsequent phases. A
phase is considered beneficial only if it achieves lower CER than Phase~2.

\subsection{Phase 3: Confusion-Matrix Injection}

Phase~3 extends the system prompt with a list of Qaari's most frequent
character confusions extracted from the Phase~1 confusion matrix. Specifically,
the top-10 character-pair confusions per dataset are formatted as an Arabic
bullet list and injected as a \texttt{<confusion\_patterns>} section before the
correction examples. The hypothesis is that telling the model which characters
Qaari tends to mix up should focus its attention on those specific pairs.

\subsection{Phase 4A: Rule-Augmented Prompting}

Arabic orthography has several rule-governed phenomena that OCR engines
frequently mishandle: the distinction between hamza al-qat' and hamza al-wasl,
the taa marbuta vs.\ terminal ha, the alef maqsura vs.\ ya, and
similar-shape letter pairs. Phase~4A injects a curated set of six rule
categories as an \texttt{<additional\_rules>} section before the correction
examples.

\subsection{Phase 4B: Few-Shot Examples from QALB}

The QALB corpus~\cite{mohit2014} provides pairs of erroneous Arabic sentences
and their corrections. Phase~4B selects five diverse pairs from the
OCR-relevant subset of QALB (filtered to pairs where errors overlap with
known OCR error types: hamza, taa marbuta, alef forms, dot confusions) and
formats them as \texttt{<additional\_examples>} before the output format
instruction. The selection strategy prioritizes error-type diversity.

\subsection{Phase 4C: CAMeL Morphological Post-Processing}

Unlike the other phases, Phase~4C does not modify the prompt. Instead,
it applies a post-processing step after zero-shot correction: each word in the
LLM output is checked against the CAMeL Tools morphological
analyzer~\cite{obeid2020}. If a corrected word fails morphological analysis
\textit{and} the original OCR word passes it, the corrected word is reverted
to the OCR original. The strategy is conservative by design---it only
intervenes when the LLM has produced a morphologically invalid word from a
valid source.

\subsection{Phase 4D: Self-Reflective Prompting}

Phase~4D feeds the model a statistical summary of its own past mistakes. To
generate this summary, Phase~2 is run on the training split, and the
corrections are compared to ground truth by an error analyzer that tracks the
fix rate and introduction rate per error type. The insights are aggregated
separately for PATS-A01 and KHATT (reflecting their different error profiles)
and formatted as Arabic-language diagnostic notes injected in a
\texttt{<self\_analysis>} section.

Additionally, concrete word-level OCR error pairs (15 per sample) extracted
from the training split are included, showing the model specific examples of
\textit{ocr\_word}~$\to$~\textit{gt\_word} corrections it should apply. This
combines statistical self-awareness with concrete exemplars.

\subsection{Phase 5: Combinations and Ablation}

Phase~5 systematically tests combinations of the individual strategies from
Phases~3, 4A, 4B, and~4D. Each combination is defined by a four-tuple
indicating which knowledge sources are active (confusion matrix, rules,
few-shot examples, self-reflective insights). This yields pairwise
combinations, triple combinations, and the full system with all four sources
active.

Ablation analysis starts from the full system and removes one component at a
time, measuring the CER impact of each removal to determine each component's
marginal contribution. CAMeL post-processing (Phase~4C) is tested as an
orthogonal post-processing layer on top of the best-performing prompt
combinations.

\subsection{Phase 7: DSPy Automated Prompt Optimization}

Phase~7 uses DSPy~\cite{dspy2024} to automatically discover optimized prompts.
A training set of 50 OCR--ground-truth pairs and a development set of 20 pairs
are drawn from the training splits. DSPy's BootstrapFewShot optimizer compiles
a declarative correction program into an optimized prompt by selecting the most
effective few-shot demonstrations from bootstrapped traces. The compiled
program is then used for inference on the validation splits, allowing direct
comparison with the hand-crafted strategies.

failures, stroke ambiguities, and writer-specific variations that are harder

QALB-derived correction examples yield \tbd\ performance. The QALB examples

fonts differently depending on which error types dominate.

Morphological validation produces \tbd\ results. Given that OCR errors are

insights are computed separately for PATS-A01 and KHATT, reflecting their

\subsection{Summary}

The results paint a picture where correction effectiveness depends on the
interaction between font characteristics, baseline error rates, and prompt
strategy. The key patterns are:

\begin{enumerate}
\item Zero-shot correction helps most when baseline CER is moderate (roughly
      10--25\%), where the LLM can identify and fix errors without excessive
      intervention.
\item On low-error fonts, LLM correction risks over-correction: the model
      rewrites tokens that Qaari rendered correctly.
\item On very high-error input (KHATT), the errors are too diverse and
      structurally different for text-only LLM correction to resolve.
\item Knowledge augmentation shows \tbd\ patterns across fonts, with no single
      strategy universally dominating.
\end{enumerate}

% -------------------------------------------------------------------------
\section{Discussion}
% -------------------------------------------------------------------------

\subsection{The Over-Correction Threshold}

The font-dependent results suggest an \textit{over-correction threshold}:
there exists a baseline CER range where LLM correction is beneficial, below
which the model introduces more errors than it fixes, and above which the
errors are too structurally complex for text-level correction. For PATS-A01
fonts, training-split analysis suggests this beneficial range is approximately
10--25\% CER. Fonts below this range (e.g., Akhbar at $\sim$4\% CER) are
harmed by any LLM intervention, while fonts within the range (Traditional,
Naskh, Thuluth) show clear improvement.

\subsection{Why Typewritten and Handwritten Differ}

KHATT's marginal improvement despite its high baseline error rate is striking.
The explanation lies in the nature of the errors: typewritten OCR errors are
predominantly character-level substitutions (dot confusions, similar shapes)
that preserve word structure, making them recoverable from text context alone.
Handwritten OCR errors include character merges, splits, segmentation
failures, and stroke misinterpretation that fundamentally alter word structure.
These errors often produce output that does not resemble any Arabic word,
giving the LLM little to work with.

This finding implies that effective handwritten OCR correction may require
multi-modal approaches that combine the OCR image with the text output, rather
than text-only post-processing.

\subsection{Prompt-Induced Over-Correction}

When augmentation degrades performance, the mechanism is consistent across
strategies: additional context raises the model's threshold for leaving text
unchanged. A zero-shot prompt with a conservative instruction (``return as-is
if correct'') keeps the model in a low-intervention mode. Confusion lists,
rules, examples, and diagnostic notes all signal that something is wrong with
the input and that active correction is expected. The model obliges, rewriting
portions of text that Qaari had got right. This is particularly costly on
low-error fonts where most tokens are already correct.

\subsection{Why CAMeL Post-Processing Has Limited Effect}

The morphological revert strategy's limited impact follows directly from the
error taxonomy: OCR errors are predominantly valid-but-wrong (real Arabic words
that are incorrect in context), not non-words. A context-free morphological
check cannot distinguish a correct word from an incorrect one if both are
morphologically valid. More powerful approaches---context-aware disambiguation
or semantic validation---might activate on more cases but would also risk
introducing their own errors.

\subsection{Practical Implications}

These results suggest a practical deployment strategy: rather than applying
a single LLM correction pipeline uniformly, systems should estimate the
baseline OCR quality and select the correction strategy accordingly.
High-quality OCR output should be left unchanged or corrected only with the
most conservative strategy. Moderate-error output benefits from zero-shot
correction. High-error handwritten output may require fine-tuned models or
multi-modal correction rather than prompt engineering alone.

\subsection{Limitations}

This study tests one LLM (Qwen3-4B) and one OCR engine (Qaari). Larger
models may exhibit different over-correction patterns. The PATS-A01 dataset
uses synthetic document images; real-world typewritten documents may introduce
additional noise (scanning artifacts, paper degradation) that changes the
error profile. The evaluation uses CER and WER as proxies for downstream
task performance; domain-specific metrics may tell a different story.

% -------------------------------------------------------------------------
\section{Conclusion}
% -------------------------------------------------------------------------

We evaluated seven prompt-engineering strategies for Arabic post-OCR correction
across nine datasets spanning eight typewritten fonts and one handwritten
corpus. The results reveal that correction effectiveness depends strongly on
the baseline error profile. Zero-shot LLM correction reduces CER substantially
on typewritten fonts with moderate-to-high error rates (up to \tbd\% relative
improvement on Traditional font) but increases errors on fonts where Qaari
already performs well. On handwritten text, improvement is marginal despite
high baseline errors.

No single knowledge-augmentation strategy dominates across all datasets.
The font-dependent behavior we observe---invisible in single-dataset
studies---highlights the importance of multi-dataset evaluation for post-OCR
correction research.

The practical implication is that effective OCR correction requires
error-profile-aware strategy selection rather than a one-size-fits-all
approach. Future work should explore adaptive correction that adjusts its
intervention level based on estimated input quality, and multi-modal
approaches that combine visual and textual signals for handwritten text
correction.
